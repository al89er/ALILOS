# Phase 4D Manual-Confirm Attendance Design

## 1. Purpose

Phase 4D defines the safe manual-confirm attendance architecture before any attendance action is implemented.

The intended model remains:

```text
Detect -> Prepare -> Notify -> Require confirmation -> Execute
```

This phase is design only. It does not implement execution, clicking, form filling, login automation, Telegram command handling, or any attendance action API.

## 2. Current State

The app currently supports:

- Starting and stopping a Playwright browser manually.
- Opening the configured Perakam dashboard manually.
- Read-only Perakam URL, title, page-status, and navigation-status detection.
- Read-only attendance control availability detection.
- `a50` detection for clock-in / Masa Hadir.
- `a51` detection for clock-out / Masa Keluar.
- Duplicate `a50` / `a51` handling where hidden sidebar candidates are ignored in favor of visible dashboard tile candidates.

The app currently does not support:

- Attendance clicking.
- Form filling.
- Auto login.
- Keep-alive behavior.
- Attendance execution through IPC, Telegram, or any other channel.

## 3. Non-Goals For Phase 4D

Phase 4D does not include:

- Playwright click or attendance action implementation.
- Attendance execution IPC or preload API.
- Telegram confirmation command handling.
- Login form filling.
- Auto login.
- Keep-alive behavior.
- Supabase, backend, or mobile heartbeat integration.
- SQLite.
- React.
- Any use of legacy keep-alive IDs `a56` or `a57`.
- `/clockin` or `/clockout` commands.

## 4. Manual-Confirm Concept

A confirmation is a short-lived user approval for exactly one attendance action:

- `clock-in`
- `clock-out`

Each confirmation request must be bound to:

- Local date key, for example `YYYY-MM-DD`.
- Action type, either `clock-in` or `clock-out`.
- Generated schedule time for that action.
- Current scheduler status at prompt creation.
- Perakam status snapshot at prompt creation.
- Detected visible control availability at prompt creation.
- Sanitized current Perakam URL.
- Confirmation expiry timestamp.
- A single-use confirmation token or internal confirmation ID.

The renderer must not be trusted to invent a confirmation request. A confirmation request should be generated by the main process or a main-owned service after a readiness check.

## 5. Ready-To-Confirm Rules

The app may show a confirmation prompt only when all of these are true:

- Today is not a weekend.
- Today is not skipped.
- Scheduler status for the action is `due-now` or `within-grace-period`.
- Browser state is `running`.
- Perakam page has been manually opened.
- Perakam status is `reachable` or `likely-logged-in`.
- The current sanitized page URL is on the configured Perakam host.
- No unresolved browser error is present.
- No unresolved Perakam navigation or status error is present.
- The correct action control is available.
- Clock-in requires a visible, enabled, actionable `a50` / Masa Hadir candidate.
- Clock-out requires a visible, enabled, actionable `a51` / Masa Keluar candidate.
- Hidden sidebar duplicate candidates are ignored.
- The same date/action has not already been completed.
- The request was created by app-owned readiness logic, not arbitrary renderer input.

If any rule fails, the app should report a clear not-ready reason and should not create a confirmation token.

## 6. Confirmation Expiry

Default confirmation expiry should be:

- 60 seconds after prompt creation, or
- earlier when the current grace period ends,

whichever comes first.

Expired confirmations must not be reusable. After expiry, the app must perform a fresh readiness check before offering another confirmation.

## 7. GUI-First Strategy

The first future implementation should use GUI confirmation only.

Recommended approach:

- Show readiness state on the dashboard.
- Offer a GUI confirmation prompt only when readiness rules pass.
- Require a deliberate user click on a confirm button in the Electron dashboard.
- Keep Telegram confirmation deferred until the local GUI workflow is proven safe.

GUI-first reduces risk because it avoids production Telegram command collisions and keeps confirmation tied to the active desktop session and visible app state.

## 8. Telegram Confirmation Strategy For Later

Telegram confirmation is design-only for now.

If added later, commands must remain prefixed:

- `/alilos_confirm_in <token>`
- `/alilos_confirm_out <token>`
- `/alilos_cancel <token>`

Telegram confirmation tokens should be:

- Short-lived.
- Single-use.
- Bound to date, action type, generated schedule time, and Perakam state.
- Accepted only from the configured chat ID.
- Redacted in logs if sensitive.

No unprefixed production commands should be added. No `/clockin` or `/clockout` commands should be added.

## 9. Suggested UI Wording

Ready to confirm clock-in:

```text
Clock-in is ready for confirmation. Perakam shows the Masa Hadir control, and the current schedule is within the allowed window.
```

Ready to confirm clock-out:

```text
Clock-out is ready for confirmation. Perakam shows the Masa Keluar control, and the current schedule is within the allowed window.
```

Not ready because browser is not running:

```text
Not ready: browser is not running.
```

Not ready because Perakam is not opened:

```text
Not ready: Perakam dashboard is not opened.
```

Not ready because today is skipped or weekend:

```text
Not ready: today is skipped or is a non-working day.
```

Not ready because control is unavailable:

```text
Not ready: the required Perakam control is not visibly available.
```

Confirmation expired:

```text
Confirmation expired. Run a fresh readiness check before confirming again.
```

Confirmation cancelled:

```text
Confirmation cancelled. No attendance action was performed.
```

Future action completed:

```text
Attendance action completed for this date and action.
```

## 10. Audit Logging Design

Future phases should log these events:

- Confirmation offered.
- Confirmation accepted.
- Confirmation cancelled.
- Confirmation expired.
- Confirmation rejected due to stale state.
- Future execution started.
- Future execution succeeded.
- Future execution failed.

Logs should include:

- Date key.
- Action type.
- Confirmation state.
- Sanitized Perakam URL host.
- High-level browser and Perakam state.
- High-level control availability.

Logs should not include:

- Full Telegram confirmation tokens.
- Bot tokens.
- Chat secrets.
- Full page body text.
- Login credentials.
- Raw form values.

## 11. Safety Checks Before Any Future Click

Before any future phase performs one attendance click, it must re-check:

- Action type.
- Local date key.
- Confirmation token or confirmation ID validity.
- Confirmation expiry.
- Single-use confirmation state.
- Generated schedule time.
- Scheduler status is still `due-now` or `within-grace-period`.
- Today is not skipped.
- Today is not a weekend.
- Browser is still running.
- Perakam page is still open.
- Current URL is on the configured Perakam host.
- No unresolved browser error exists.
- No unresolved Perakam error exists.
- Correct visible control candidate still exists.
- Clock-in uses visible `a50` / Masa Hadir.
- Clock-out uses visible `a51` / Masa Keluar.
- Hidden sidebar duplicate is ignored.
- No stale confirmation or already-completed state exists for the same date/action.

Only after all checks pass may a future phase perform one guarded click.

## 12. Proposed Future Phases

These are proposals only. Phase 4D does not implement them.
